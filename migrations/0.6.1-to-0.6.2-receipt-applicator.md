# 0.6.1 to 0.6.2 - separate receipt applicator

## Summary

PAM 0.6.2 separates curator judgment from canonical application. Curator
decisions now end as `review_required`, `rejected`, or
`approved_pending_apply`. The curator never calls proposal/apply and never
attests `promoted`.

## Capability migration

- Replace curator capabilities with `memory:curate`.
- Provision a different token and allowlist entry with
  `memory:apply-receipt` under `amfApplicator`.
- Do not reuse the curator credential for the applicator.

Existing `apply: true` review requests and `recover-apply` curator requests are
invalid. Record approval first, then call `memory_receipt_apply` with the
decision ID and a stable application idempotency key.

## Recovery and transport

Application state is durable under `memory/amf/applicator/` and advances from
`prepared` through PAM apply and receipt queueing to Fabric ACK. Retry the exact
application after a crash. The default transport is disabled; HTTP dispatch is
fail-closed until an audited Fabric endpoint/sink is integrated.

The Git writer remains disabled and dry-run-only.
