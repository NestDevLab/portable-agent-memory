# PAM 0.6.2 → 0.6.3: Fabric promotion transport

PAM 0.6.3 binds curator candidates to the immutable Fabric `proposalId` and
canonical proposal digest, adds bounded HTTPS intake/receipt delivery with
separate curator and applicator token files, and maintains
`memory/amf/record-index.json` atomically after apply.

Receipts bind the canonical proposal scope and digest. Fabric must authorize
that scope against the current actor ACL before reading or advancing a proposal.
Decision replay uses an authenticated persistent cursor and bounded circular
scan so terminal ACK entries cannot starve queued work.

An opt-in Git adapter can commit only the applied record and index and can push
only when repository, branch, remote, fast-forward ancestry, and explicit
`--push` all pass. It never force-pushes and is disabled by default.
Authenticated Git state binds repository identity, branch, remote URL digest and
target ref; changing any of them requires explicit recovery rather than retry.

Before enabling automatic promotion:

1. provision an external owner-only `0600` workspace config with reviewer and
   applicator token hashes;
2. provision distinct `0600` Fabric bearer-token files for the curator and
   applicator plus a separate applicator state-verification key;
   each file's parent directory must be owner-owned and not writable by group or
   others because private files are opened through a no-follow parent dirfd;
3. configure an HTTPS Fabric origin and leave redirects disabled;
4. drain one proposal without dispatch, inspect the decision outbox, then
   repeat with authenticated dispatch;
5. verify the generated record index and Fabric hot reload before increasing
   the bounded page and page-count limits.

Rollback disables the worker first. Preserve curator/applicator state, outboxes,
canonical records, and the record index. Repoint Fabric to the pre-change
read-only index snapshot; do not delete receipts or rewrite promoted records.
