# Migration: PAM 0.2.0 to 0.3.0 OpenClaw specialization

Status: optional adoption guide  
From: `0.2.0`  
To: `0.3.0`  
Graph schema change: none

## Summary

PAM 0.3.0 keeps the same graph data format as 0.2.0:

```json
{
  "memoryFormat": "graph-v1",
  "graphSchemaVersion": "pam-graph-v1"
}
```

Existing graph files remain valid. This migration is procedural: it helps a
workspace adopt the new OpenClaw specialization and installation acceptance
criteria.

## When to use this guide

Use this guide when:

- an existing PAM 0.2.0 workspace wants the OpenClaw specialization;
- an agent needs explicit installation acceptance criteria;
- a workspace wants local agent instructions to point future agents to PAM;
- runtime-native maintenance automation should be offered or configured.

Skip this guide when:

- the workspace is not OpenClaw-like and does not need the OpenClaw profile;
- the existing PAM graph works and no installation-policy changes are desired.

## Steps

1. Update `memory/pam.version.json`:

```json
{
  "pamVersion": "0.3.0",
  "memoryFormat": "graph-v1",
  "graphSchemaVersion": "pam-graph-v1"
}
```

2. Add or update the OpenClaw specialization:

```text
memory/agent-memory/pam-openclaw.md
```

3. Update the local agent instruction file, such as:

```text
AGENTS.md
CLAUDE.md
GEMINI.md
```

It should tell future agents:

- where the PAM entrypoint is;
- where the runtime guide is;
- where graph files live;
- which memory sources remain canonical;
- which paths PAM may write by default.

4. Verify installation acceptance criteria.

The final response should report each criterion with colored status markers:

- 🟢 `PASS` — complete and verified;
- 🟡 `PARTIAL` — partly complete; explain the gap;
- 🔵 `DEFERRED` — intentionally postponed or opted out;
- 🔴 `BLOCKED` — cannot proceed safely;
- ⚪ `N/A` — not applicable.

5. Offer runtime-native maintenance automation.

For OpenClaw, the recommended automation is an OpenClaw cron job scoped to
PAM-owned paths such as:

```text
memory/graph/*
memory/maintenance/*
```

Automation is strongly recommended but opt-in. If the user declines or defers it,
record that as `DEFERRED`, not as a failed migration.

## Safety rules

Do not rewrite canonical runtime/project memory by default.

For OpenClaw, do not rewrite these without explicit user approval:

```text
MEMORY.md
memory/**/*.md existing corpus files
compiled wiki pages
project-specific conventions
```

## Validation

Run graph validation when available:

```bash
npm run memory:graph:validate
```

If Node tooling is unavailable, validate JSON/JSONL structure manually and state
that tool validation was not available.

## Expected result

A successful adoption has:

- unchanged graph schema (`pam-graph-v1`);
- updated PAM version metadata;
- OpenClaw specialization installed when applicable;
- local agent instructions updated;
- acceptance criteria reported criterion by criterion;
- maintenance automation configured or explicitly deferred.
